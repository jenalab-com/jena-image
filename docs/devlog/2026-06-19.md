## #1 동시성 갭 제거 — UI 블로킹 해소

### 배경
설계 문서(`docs/architecture.md` §8)는 "모든 이미지/파일 I/O는 백그라운드"를 표방하나,
실제 구현은 그렇지 않았다.
- `ImageService.loadImage`/`generateThumbnail`은 `async` 선언만 돼 있고 내부에
  suspension point가 없어, `@MainActor` 호출자가 `await`하면 본문이 **호출자(메인)
  스레드에서 그대로 실행**되었다. (Swift `async`의 함정 — 키워드만으론 백그라운드로 안 감)
- `FileService.contentsOfFolder`는 동기 함수라 대용량 폴더 열거가 메인 스레드를 막았다.

### 변경 사항
- [Sources/services/ImageService.swift](../../Sources/services/ImageService.swift)
  - `loadImage`/`generateThumbnail` 본문을 `Task.detached(priority: .userInitiated)`로
    감싸 디코딩을 실제 백그라운드에서 수행. 동기 본문은 `loadImageSync` /
    `generateImageThumbnailSync`로 분리.
  - 비디오 썸네일(`generateVideoThumbnail`)은 이미 `AVAssetImageGenerator`로 진짜
    비동기라 그대로 유지.
  - stateless 클래스이므로 `@unchecked Sendable`을 채택해 백그라운드 Task에서 안전 캡처.
  - 호출부(MainWindowController, ViewerViewController, BrowserViewController,
    ThumbnailStripView) 4곳은 이미 `await` 호출이라 **변경 없음**.
- [Sources/app/MainWindowController.swift](../../Sources/app/MainWindowController.swift)
  - `navigateToFolder`, `refreshAfterExternalChange`의 디렉토리 열거를
    `Task.detached`로 백그라운드 수행 후 `@MainActor`에서 UI 갱신.
  - **stale 가드** 추가: 열거 완료 시점에
    `currentFolderURL == url && contentMode == .browser`를 확인해, 빠른 폴더 전환이나
    뷰어 진입 후 늦게 끝난 열거가 화면을 덮어쓰지 않게 함.
    (architecture.md §7.2 "비동기 작업 취소" 원칙과 합치)

### 트레이드오프 / 남긴 것
- `imageFilesInCurrentFolder()`(뷰어 진입 1회성)는 여전히 동기 열거. 단발이라 이번
  범위 밖. 추후 browser가 이미 보유한 목록을 재사용하면 디스크 재열거까지 제거 가능.
- 사이드바에서 파일 직접 선택→뷰어 경로(`didSelectImage`)는 `navigateToFolder`
  비동기화 후 미묘함이 있어 stale 가드로 방어. 수동 테스트 권장 지점.

## #2 MainWindowController 분해 — 1063줄 → 코어 366줄

### 배경
`MainWindowController`가 윈도우 생명주기·네비게이션·파일 오퍼레이션·툴바·메뉴 검증·
4개 델리게이트를 모두 들고 있는 God object 상태. ADR-004가 예견한 비대화가 현실화됨.
이번엔 **extension 파일 분할**(점진적·동작 보존)까지만 진행. 진짜 책임 분리(별도 타입
추출)는 다음 단계로 남김.

### 변경 사항
- [Sources/app/MainWindowController.swift](../../Sources/app/MainWindowController.swift)
  (코어 366줄) — enum·프로퍼티·init·생명주기·Folder Access·Navigation·Folder
  Watcher·Setup·공통 헬퍼.
- [Sources/app/MainWindowController+FileOperations.swift](../../Sources/app/MainWindowController+FileOperations.swift)
  (196줄) — performDelete/Move/MoveToFolder/Copy/Rename/Export, 충돌 처리.
- [Sources/app/MainWindowController+Actions.swift](../../Sources/app/MainWindowController+Actions.swift)
  (314줄) — 툴바·메뉴 @objc 액션, 활성 선택 헬퍼, `validateMenuItem`.
- [Sources/app/MainWindowController+Delegates.swift](../../Sources/app/MainWindowController+Delegates.swift)
  (228줄) — NSToolbarDelegate / Sidebar / Browser / Viewer 델리게이트.

### 판단 근거
- Swift `private`는 파일 스코프이므로, extension을 다른 파일로 옮기면 공유 멤버와
  `#selector`로 참조되는 `@objc` 액션이 안 보여 컴파일 실패. 따라서 여러 파일이
  공유하는 멤버를 `private` → `internal`(기본)로 완화.
- 단일 앱 모듈이라 `internal` 노출은 외부에 새지 않으며, 접근 수준 완화는 동작을
  바꾸지 않음(컴파일 타임 가시성만). 파일 내부 전용 헬퍼(`handleFileConflict`,
  `generateUniqueName`, 셋업/디바운스 등)는 `private` 유지.

## #3 이미지 에디터 undo 스택 메모리 폭증 방지

### 배경
[Sources/viewer/ImageEditorWindowController.swift](../../Sources/viewer/ImageEditorWindowController.swift)의
undo/redo가 풀해상도 `NSImage`를 **개수(10장) 제한만**으로 보관. 50MP급 이미지면
한 장당 수백 MB라, 편집 히스토리가 메모리를 폭증시킬 수 있었다.

### 변경 사항
- `maxHistoryBytes`(384MB) 예산 도입.
- `pushUndo` → `trimHistory` 분리: 개수 상한 + 메모리 예산 초과 시 가장 오래된
  항목부터 제거. 단 직전 동작은 되돌릴 수 있게 **최소 1장은 보존**.
- `estimatedBytes(of:)` — `pixelsWide × pixelsHigh × 4`(RGBA) 추정.
  (ThumbnailCache의 cost 산정 방식과 동일)

### 판단 근거
- 개수 제한만으론 이미지 크기에 따른 메모리 편차를 못 잡음. 예산 기반이면 작은
  이미지는 깊은 히스토리, 큰 이미지는 얕은 히스토리로 자동 조절됨.
- NSImage를 그대로 보관하므로 압축/복원 디코드 비용 없음.

## 뷰어 확장 방향 결정 + 정렬(1순위) 구현

### 방향 결정
- 편집기를 포토샵급(레이어·마스크·합성·레벨·커브 등)으로 키우는 안을 brainstorming.
  결론: **편집기 확장은 [docs/editor-roadmap.md](../editor-roadmap.md)에 4단계 계획으로
  박제하고 미착수.** 비파괴 편집 파이프라인 재설계가 토대라 짜투리 프로젝트엔 과한
  삽질. 먼저 뷰어를 강화하기로.
- 뷰어 기능 우선순위 확정(1차: 정렬·QuickLook·슬라이드쇼·복제). 1순위 정렬부터 착수.

### 정렬 기능
브라우저 그리드 정렬 기준을 이름순 고정에서 **이름·날짜·크기·종류 + 오름/내림**으로 확장.
정렬 상태는 앱 재시작 후에도 유지.

### 변경 사항
- [Sources/services/FileService.swift](../../Sources/services/FileService.swift)
  - `SortKey` enum(name/date/size/kind) 추가.
  - `contentsOfFolder(at:sortKey:ascending:)`로 시그니처 확장. 열거 시
    `.contentModificationDateKey`·`.fileSizeKey`를 prefetch.
  - `sortURLs(_:by:ascending:)` 추가 — 정렬 기준 값을 **URL당 한 번만** 읽어
    딕셔너리에 캐시한 뒤 정렬(비교 중 중복 디스크 접근 방지). 동률은 이름순 안정화,
    내림차순은 결과 reverse. 폴더/이미지 각각 정렬(폴더 항상 위).
- [Sources/services/AppSettings.swift](../../Sources/services/AppSettings.swift)
  - `sortKey`, `sortAscending` 영속 프로퍼티 추가(UserDefaults).
- [Sources/app/MainWindowController.swift](../../Sources/app/MainWindowController.swift)
  - `ToolbarID.sort` 추가, `sortMenu`(lazy, delegate=self) 보관.
  - `navigateToFolder`·`refreshAfterExternalChange`·`imageFilesInCurrentFolder`
    호출부에 현재 정렬 설정 전달(백그라운드 열거 진입 전 메인에서 읽어 캡처).
- [Sources/app/MainWindowController+Actions.swift](../../Sources/app/MainWindowController+Actions.swift)
  - `NSMenuDelegate` 채택. `menuNeedsUpdate`에서 정렬 메뉴를 열 때마다 현재 상태
    (체크 표시)로 재구성.
  - `sortByMenuItem`(같은 기준 재선택 시 방향 토글 — Finder 동작), `toggleSortOrder`.
- [Sources/app/MainWindowController+Delegates.swift](../../Sources/app/MainWindowController+Delegates.swift)
  - 툴바에 `NSMenuToolbarItem`(정렬 팝업, `arrow.up.arrow.down`) 추가.

### 판단 근거
- 정렬을 `contentsOfFolder`에서 처리 → 모든 호출부(navigate/refresh/뷰어목록)가
  일관된 순서를 받음. `BrowserViewController.display`는 받은 순서대로 표시하므로
  브라우저 코드는 무변경.
- 정렬 기준 값 prefetch + 딕셔너리 캐시로 대용량 폴더에서도 정렬 비용 O(n) 유지.
- UI는 툴바 팝업(Finder/미리보기와 동일한 위치감) 채택.

### 검증

## QuickLook 미리보기 (2순위) 구현

브라우저 그리드에서 **스페이스바로 QuickLook 미리보기**를 띄운다. 패널 안에서
←→로 폴더 전체를 순회하고, 스페이스로 닫는다. (Finder 동작과 동일)

### 변경 사항
- [Makefile](../../Makefile): `-framework Quartz` 추가 (QLPreviewPanel).
- [Sources/browser/BrowserViewController.swift](../../Sources/browser/BrowserViewController.swift)
  - `import Quartz`.
  - `DoubleClickCollectionView`: `onSpaceKey` 콜백 + `keyDown` 오버라이드(keyCode 49).
  - `toggleQuickLook()` — 패널 토글. 빈 폴더면 무시.
  - `QLPreviewPanelController` 제어 메서드 오버라이드
    (`acceptsPreviewPanelControl`/`begin`/`end`) — NSResponder 체인으로 패널 제어권 획득.
    `beginPreviewPanelControl`에서 현재 선택 항목을 시작 인덱스로 설정.
  - `QLPreviewPanelDataSource` — `contents` 전체를 미리보기 항목으로 제공
    (URL→`NSURL`은 `QLPreviewItem` 준수). 단일 선택이어도 패널 안에서 폴더 전체 순회 가능.
  - `QLPreviewPanelDelegate` — 패널이 떠 있을 때 스페이스로 닫기, ←→는 기본 순회에 위임.

### 판단 근거
- QuickLook은 responder 체인 기반이라, 컬렉션뷰의 nextResponder인
  `BrowserViewController`에 제어 메서드를 두면 자연스럽게 제어권을 받는다.
- 미리보기 항목을 선택분이 아니라 폴더 전체(`contents`)로 주어 Finder식 순회를 지원.

### 검증
- `make build` EXIT=0, 경고·에러 0.
- **GUI 수동 테스트 필요**: 그리드에서 항목 선택 후 스페이스 → 미리보기, ←→ 순회,
  스페이스로 닫기, 폴더/비디오 항목도 미리보기 되는지 확인.
- ✅ 수동 테스트 통과 (`make run`).
